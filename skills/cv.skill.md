## Overview

Computer vision skill files generate projects that demonstrate practical vision system design: from classical image processing through deep learning inference pipelines to production model serving. CV projects are high-signal portfolio pieces because they require understanding of both ML fundamentals (model selection, training, evaluation) and systems engineering (GPU memory management, batching, latency optimization). Projects include model training scripts, evaluation harnesses, a REST inference API, and Dockerized deployment. All generated projects use pretrained model weights where possible so the sandbox can run end-to-end without requiring expensive GPU training.

## Architecture Patterns

All CV projects generated by this skill MUST follow these patterns:

1. **Model and inference are separated.** A `models/` directory contains model definitions and weight-loading logic. An `inference/` module contains batching, preprocessing, and postprocessing. Route handlers only call the inference module — they never call model objects directly.

2. **Preprocessing is reproducible and tested.** All image transforms (resize, normalize, color space conversion) are defined as versioned pipeline objects, not inline code. Tests assert that the same input image produces the same tensor representation across runs.

3. **Confidence thresholds are configuration, not code.** Detection thresholds, NMS parameters, and class filters are read from config, not hardcoded. This allows non-code changes to tune behavior.

4. **Model weights are versioned.** Weight files are tracked in a `model_registry.json` with SHA-256 checksums. The loader validates checksums before serving to catch corrupted downloads.

5. **Latency and throughput are measured.** Every project includes a `benchmark.py` script that measures p50/p95/p99 latency and max throughput for a synthetic input workload.

6. **GPU memory is managed explicitly.** For PyTorch projects, `torch.no_grad()` context is used for all inference, and batch sizes are validated against available memory before loading.

## Quality Criteria

The QualityGate checks the following for CV projects:

1. **Tests pass.** `pytest tests/ -v` exits with code 0.
2. **Linting passes.** `ruff check .` reports zero errors.
3. **Type checking passes.** `mypy --strict src/` exits with code 0.
4. **Inference pipeline runs on a sample image.** The benchmark script completes without error against a bundled test image.
5. **No model weights committed.** The `.gitignore` must exclude `.pt`, `.pth`, `.onnx`, `.bin` weight files. Weights are downloaded by `scripts/download_weights.py`.
6. **API starts and responds.** `uvicorn src.api.main:app` starts without error and responds to a health check at `/health`.
7. **Confidence threshold is configurable via environment.** No hardcoded threshold values in route handlers.

## ADR Templates

### ADR-001: Model Selection

**Status:** Accepted

**Context:** The project requires a pretrained object detection backbone. Candidates include YOLOv8, RT-DETR, and EfficientDet. The model must run on CPU for sandbox validation and be fine-tunable on a small dataset.

**Decision:** YOLOv8 (Ultralytics) for detection tasks. It provides a clean Python API, ONNX export, and strong pretrained weights that run acceptably on CPU. For segmentation tasks, SAM (Segment Anything) for zero-shot use or a pretrained U-Net when fine-tuning on a specific domain.

**Consequences:** We benefit from a well-maintained library with active community support. We accept the Ultralytics license terms (AGPL-3.0 for YOLOv8), which is acceptable for portfolio projects.

### ADR-002: Serving Architecture

**Status:** Accepted

**Context:** The inference service must handle concurrent image requests. Options include FastAPI with async handlers, Triton Inference Server, and TorchServe.

**Decision:** FastAPI with synchronous model inference running in a thread pool executor (`run_in_executor`). Triton is production-ideal but adds significant operational complexity. TorchServe requires Java. FastAPI + thread pool gives good throughput for portfolio demonstration without requiring a GPU host.

**Consequences:** We achieve clean async HTTP handling without blocking the event loop. Throughput is limited by the thread pool size (default 4). We document the Triton upgrade path in the README for production scenarios.

### ADR-003: Image Storage Strategy

**Status:** Accepted

**Context:** The API receives binary image uploads. We need a strategy for storing, caching, and garbage-collecting processed images.

**Decision:** Images are processed in-memory only (never persisted to disk) unless the project explicitly requires a corpus (e.g., similarity search). For corpus-based projects, images are stored in a local SQLite-backed object store with SHA-256 content addressing to deduplicate.

**Consequences:** In-memory processing eliminates disk I/O latency and storage concerns for the inference API. For corpus projects, content addressing ensures the same image is stored once regardless of how many times it is indexed.

## Common Failure Modes

1. **OpenCV import fails in Docker.** `cv2` requires system libraries (`libgl1-mesa-glx`, `libglib2.0-0`) that are not present in the base Python image. Fix: add these packages to `extra_packages` in frontmatter and install via `apt-get` in the Dockerfile.

2. **Model weight download fails in sandbox.** Sandbox containers have outbound-only networking, but some weight hosting services require authentication. Fix: use `download_weights.py --dry-run` to validate URLs only; actual weights are pre-downloaded into the sandbox image for heavy models.

3. **Tensor shape mismatch after preprocessing.** Models expect specific input shapes (e.g., NCHW vs NHWC). Fix: include an explicit `assert input_tensor.shape == expected_shape` in the pipeline with a descriptive error message.

4. **Float precision errors in test assertions.** Image pixel values after normalize/denormalize cycles may differ by small floating point amounts. Fix: use `torch.allclose(a, b, atol=1e-5)` instead of `==` for tensor comparisons.

5. **ONNX export fails on custom ops.** Some PyTorch custom operations are not ONNX-exportable without explicit operator definitions. Fix: use `torch.onnx.export` with `opset_version=17` and test export as part of CI.
