## Overview

Data science skills generate projects that demonstrate the full analytical workflow: data exploration, hypothesis formulation, feature engineering, modeling, evaluation, and communication of results. These projects are critical portfolio pieces for data science roles because they prove the candidate can go beyond running `.fit()` on a dataset and actually deliver actionable insights with rigorous methodology.

Every project generated by this skill includes clean, reproducible analysis with clear documentation, proper train/test splitting, statistical rigor, meaningful visualizations, and a presentable deliverable (dashboard, report, or API). The emphasis is on analytical thinking and communication, not just model accuracy. Projects demonstrate that the candidate can frame business problems as data problems, select appropriate methods, validate results, and explain findings to non-technical stakeholders.

## Architecture Patterns

All data science projects generated by this skill MUST follow these patterns:

1. **Reproducible analysis pipeline.** Every analysis step is scripted and runnable from the command line. Random seeds are fixed. Data versions are tracked. Results can be regenerated from scratch with a single command.

2. **Proper train/test separation.** Data splitting happens once, early, and is never violated. The test set is held out until final evaluation. Cross-validation is used for model selection and hyperparameter tuning on the training set only.

3. **Configuration-driven experiments.** Model parameters, feature selections, and analysis configurations live in YAML or TOML files. Experiment variations are defined by configuration, not by code changes.

4. **Statistical rigor.** Confidence intervals accompany all point estimates. Significance tests include effect sizes, not just p-values. Multiple comparison corrections are applied when testing multiple hypotheses. Assumptions are explicitly checked and documented.

5. **Visualization standards.** All charts have labeled axes, titles, and legends. Color palettes are colorblind-friendly. Interactive visualizations are preferred for dashboards. Static visualizations are high-resolution and publication-quality.

6. **Modular analysis code.** Data loading, preprocessing, feature engineering, modeling, and evaluation live in separate modules. Notebooks are for presentation only; all logic is importable from Python modules.

7. **Documentation as a deliverable.** The final output includes not just code and models, but a narrative document or dashboard that explains the analysis, methodology, findings, and recommendations in plain language.

## Quality Criteria

The QualityGate checks the following for data science projects:

1. **All tests pass.** `pytest tests/ -v` exits with code 0. Tests validate data preprocessing, feature engineering, model training, and evaluation metric computation.
2. **Linting passes.** `ruff check .` reports zero errors.
3. **No data leakage.** Code review verifies that test data is never used during training or feature selection. No information from future observations leaks into past predictions.
4. **Reproducibility.** Running the analysis pipeline twice with the same random seed produces identical results. Seeds are set in configuration, not hardcoded.
5. **No hardcoded file paths.** All data paths, output paths, and configuration paths are parameterized.
6. **Visualizations are complete.** All charts have axis labels, titles, and legends. No `plt.show()` calls in module code (only in notebooks or dashboard).
7. **README tells a story.** README includes: problem statement, data description, methodology, key findings, and instructions to reproduce.
8. **Sample data included.** A small sample dataset is included in `data/sample/` so that the pipeline can run without external data dependencies.

## ADR Templates

### ADR-001: Modeling Approach Selection

**Status:** Accepted

**Context:** We need to select a modeling approach for the prediction/analysis task. Options range from simple linear models to complex ensemble methods and deep learning.

**Decision:** We start with a simple baseline model (logistic regression or linear regression, depending on the task) and progressively add complexity. The baseline establishes a performance floor. We then compare gradient-boosted trees (XGBoost), random forests, and regularized linear models. The model with the best cross-validated performance on a held-out validation set, considering both accuracy and interpretability, is selected.

**Consequences:** We get a principled model selection process with clear justification. The baseline-first approach ensures we do not over-engineer. The comparison framework is reusable across experiments. We accept the computational cost of training multiple models, mitigated by starting with a small sample for initial comparison.

### ADR-002: Evaluation Strategy

**Status:** Accepted

**Context:** The model's performance must be evaluated rigorously before deployment or presentation. We need to decide on evaluation methodology and metrics.

**Decision:** We use stratified k-fold cross-validation (k=5) for model selection and hyperparameter tuning. Final evaluation is performed on a held-out test set (20% of data, stratified). We report multiple metrics: the primary metric (task-dependent), secondary metrics, and confidence intervals via bootstrap resampling. We use SHAP values for model interpretability.

**Consequences:** We get statistically robust performance estimates with uncertainty quantification. The held-out test set provides an unbiased final estimate. SHAP values provide interpretability regardless of model type. We accept the computational overhead of cross-validation and bootstrap resampling.

## Common Failure Modes

1. **Data leakage through preprocessing.** Fitting a scaler or encoder on the full dataset (including test data) before splitting leaks information. Fix: all preprocessing must be fit on training data only and applied (transform-only) to test data. Use sklearn Pipelines to enforce this.

2. **Missing random seed configuration.** Analysis produces different results on each run, making tests flaky. Fix: set seeds for `random`, `numpy`, and `sklearn` in a central configuration that is loaded at pipeline startup.

3. **Large data files committed to git.** Sample data files exceed git size limits or bloat the repository. Fix: `.gitignore` includes `data/raw/` and `data/processed/`. Only `data/sample/` (small files) is committed. README documents how to obtain the full dataset.

4. **Notebook-only logic.** Analysis logic lives only in Jupyter notebooks and cannot be tested or imported. Fix: all logic lives in `src/` modules. Notebooks import from `src/` and are used for presentation only.

5. **Hardcoded column names.** Feature engineering code uses hardcoded column names that break when the dataset changes. Fix: column names come from configuration files or are discovered dynamically with validation.

6. **Missing dependencies for visualization.** Plotly or matplotlib backends require system libraries (e.g., libfreetype) that are not in the sandbox. Fix: include required system packages in `sandbox_requirements.extra_packages` and test imports in the sandbox validation plan.
