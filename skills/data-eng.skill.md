## Overview

Data engineering skills generate projects that demonstrate the ability to build reliable, scalable data pipelines: ingestion, transformation, quality validation, orchestration, and delivery to downstream consumers. These are high-value portfolio pieces because data engineering roles are in heavy demand and most candidates only show SQL queries or single-script ETL jobs. A well-structured data pipeline project proves the candidate can design for idempotency, handle schema evolution, implement data quality checks, and orchestrate multi-step workflows with proper error handling and observability.

Every project generated by this skill includes orchestrated pipelines, transformation layers with clear lineage, data quality validation, configuration-driven behavior, and comprehensive testing of both pipeline logic and data contracts.

## Project Templates

### 1. ETL Pipeline with Data Quality Checks

- **Name:** dataflow-etl
- **One-liner:** An ETL pipeline that ingests CSV/JSON data, validates quality with Great Expectations, transforms with dbt, and loads into a warehouse with full lineage tracking.
- **Seniority:** junior
- **Tech Stack:** Python, dbt, DuckDB, Great Expectations, Airflow (local executor)
- **Complexity:** 3
- **Why It Impresses:** Junior data engineers who demonstrate data quality validation alongside basic ETL stand out immediately. Most junior portfolios are raw pandas scripts. This project shows understanding of pipeline orchestration, data contracts, and reproducible transformations.

### 2. Data Lakehouse with Medallion Architecture

- **Name:** lake-forge
- **One-liner:** A lakehouse implementation with bronze/silver/gold medallion layers, schema enforcement, incremental processing, and a self-service query interface.
- **Seniority:** mid
- **Tech Stack:** PySpark, Delta Lake, Airflow, dbt, FastAPI (query API)
- **Complexity:** 6
- **Why It Impresses:** The medallion architecture is the industry standard for modern data platforms. Building one demonstrates understanding of data quality tiers, incremental vs. full refresh strategies, and the tradeoffs between raw data preservation and curated analytical datasets. The query API layer shows the candidate thinks about data consumers, not just pipelines.

### 3. Real-time Stream Processing Platform

- **Name:** stream-engine
- **One-liner:** A real-time stream processing platform with event ingestion, windowed aggregations, late data handling, dead letter queues, and exactly-once delivery guarantees.
- **Seniority:** senior
- **Tech Stack:** Python, Apache Kafka (or Redpanda), Flink (or Spark Structured Streaming), PostgreSQL, Prometheus, Grafana
- **Complexity:** 8
- **Why It Impresses:** Real-time data processing is significantly harder than batch ETL. This project demonstrates understanding of event time vs. processing time, watermarks, windowing strategies, and exactly-once semantics. The dead letter queue and late data handling show production-grade thinking that separates senior engineers from mid-level.

## Architecture Patterns

All data engineering projects generated by this skill MUST follow these patterns:

1. **Idempotent pipelines.** Every pipeline step must be safely re-runnable. Insert operations use upsert or merge semantics. Transformations produce the same output given the same input regardless of how many times they run.

2. **Configuration-driven behavior.** Pipeline schedules, source connections, transformation parameters, and quality thresholds are all defined in configuration files (YAML or TOML). No hardcoded connection strings or magic constants.

3. **Data quality as a first-class citizen.** Every pipeline includes at least schema validation (column types, nullability) and semantic validation (value ranges, referential integrity). Quality failures halt downstream processing and produce structured alerts.

4. **Clear data lineage.** Every transformation documents its inputs, outputs, and business logic. dbt models include descriptions and tests. Pipeline DAGs have meaningful task names and documentation strings.

5. **Separation of orchestration and logic.** Airflow DAGs (or equivalent) contain scheduling and dependency logic only. Business transformation logic lives in separate Python modules or dbt models that can be tested independently.

6. **Schema evolution handling.** Pipelines gracefully handle additive schema changes (new columns) without failure. Breaking schema changes are detected and surfaced as structured errors before data corruption occurs.

7. **Environment parity.** Local development, CI, and production use the same transformation logic. Only connection strings and scale differ between environments.

## Quality Criteria

The QualityGate checks the following for data engineering projects:

1. **Pipeline tests pass.** `pytest tests/ -v` exits with code 0. At least one test per pipeline stage (ingestion, transformation, quality validation).
2. **dbt models compile.** `dbt compile` succeeds with no errors. All models have at least one test defined in `schema.yml`.
3. **Linting passes.** `ruff check .` reports zero errors for Python code. `sqlfluff lint` reports zero errors for SQL files (if sqlfluff is in dependencies).
4. **No hardcoded connection strings.** All database URIs, API endpoints, and credentials come from environment variables or config files.
5. **Idempotency is tested.** At least one test verifies that running a pipeline step twice produces the same result.
6. **Data quality checks exist.** At least 3 data quality assertions are defined and tested.
7. **README documents the pipeline.** README includes a DAG diagram (ASCII or Mermaid), setup instructions, and a description of each pipeline stage.
8. **Configuration is validated.** Config files are loaded and validated against schemas at startup, not at runtime deep in the pipeline.

## ADR Templates

### ADR-001: Orchestration Framework Choice

**Status:** Accepted

**Context:** We need a workflow orchestration framework to schedule and monitor data pipelines. The primary candidates are Apache Airflow, Prefect, and Dagster.

**Decision:** We chose Apache Airflow because it is the industry standard for data pipeline orchestration, has the largest community, and is the most commonly requested skill in data engineering job descriptions. Its DAG-based model maps naturally to data pipeline dependencies.

**Consequences:** We gain a mature, battle-tested orchestration platform with extensive operator libraries and a built-in web UI for monitoring. We accept the complexity of Airflow's architecture (scheduler, webserver, worker, database) and mitigate it with Docker Compose for local development.

### ADR-002: Storage Format Selection

**Status:** Accepted

**Context:** The pipeline needs a storage format for intermediate and final datasets. Options include CSV, Parquet, Delta Lake, and Iceberg.

**Decision:** Apache Parquet for intermediate storage, with Delta Lake for the lakehouse layer (when applicable). Parquet provides columnar compression, schema enforcement, and broad tool compatibility. Delta Lake adds ACID transactions, time travel, and merge operations on top of Parquet.

**Consequences:** We get efficient columnar storage with strong schema enforcement and broad ecosystem support. Parquet files are readable by Spark, DuckDB, pandas, and Polars without conversion. Delta Lake adds transactional guarantees at the cost of a thin metadata layer.

## Common Failure Modes

1. **Airflow DAG import errors.** DAG files that import from the `pipelines/` package fail if the package is not installed in the Airflow environment. Fix: ensure `pip install -e .` is run in the Airflow worker environment and the project root is on `PYTHONPATH`.

2. **DuckDB vs. PostgreSQL SQL dialect mismatch.** Tests using DuckDB as a local substitute for PostgreSQL/Snowflake encounter syntax differences. Fix: write transformation SQL in ANSI-compatible syntax and test dialect-specific features separately.

3. **Timezone handling in timestamps.** Pipelines that mix timezone-naive and timezone-aware timestamps produce incorrect results. Fix: all timestamps must be UTC with explicit timezone information. Validate in data quality checks.

4. **Missing test data fixtures.** Pipeline tests need sample data to run. Fix: include small CSV/JSON fixtures in `tests/fixtures/` and load them in `conftest.py`. Never depend on external data sources in unit tests.

5. **dbt profile misconfiguration.** dbt defaults to looking for `~/.dbt/profiles.yml` which does not exist in the sandbox. Fix: use `DBT_PROFILES_DIR` environment variable to point to the project's `profiles.yml`.

6. **Non-deterministic ordering in pipeline output.** Spark and other distributed frameworks do not guarantee row ordering. Fix: always include explicit `ORDER BY` in final output queries and test only set equality for intermediate results.
