## Overview

Mobile development skill files generate cross-platform mobile applications using React Native (Expo) or Flutter that demonstrate proficiency with mobile-specific concerns: navigation, offline-first data sync, push notifications, device API integration, and app store deployment preparation. Mobile projects are strong portfolio signals because few backend engineers are comfortable in the mobile ecosystem, making a well-executed mobile project a differentiator. Generated projects include TypeScript-typed components, a FastAPI or Supabase backend, offline support with SQLite/MMKV, and CI/CD configured for Expo EAS Build or Flutter build artifacts.

## Architecture Patterns

All mobile projects generated by this skill MUST follow these patterns:

1. **Offline-first data layer.** All mobile apps use a local SQLite or MMKV store as the primary data source. Remote sync is additive. The app is fully functional without network connectivity (degraded but usable).

2. **Navigation is typed.** React Native projects use `expo-router` (file-based routing) or `react-navigation` with a fully typed `RootStackParamList`. No `any` types in navigation prop types.

3. **Platform abstractions for device APIs.** Device-specific APIs (camera, location, biometrics, notifications) are wrapped in a platform abstraction module (`src/services/`) that returns typed results. This enables easy mocking in tests and swapping implementations.

4. **State management is layered.** Local UI state uses `useState`/`useReducer`. Server cache state uses React Query (`@tanstack/react-query`). Global app state (auth, settings) uses Zustand or Context with typed slices.

5. **Bundle size is monitored.** A `scripts/analyze-bundle.sh` script that runs `expo export --dump-sourcemap` and reports bundle size is included. The README documents the baseline bundle size.

6. **App icons and splash screen are configured.** `app.json` includes proper iOS and Android icon configurations. A placeholder app icon and splash screen are included so the app installs without Expo defaults.

## Quality Criteria

The QualityGate checks the following for mobile projects:

1. **TypeScript compiles without errors.** `tsc --noEmit` exits with code 0.
2. **Linting passes.** `eslint . --ext .ts,.tsx` reports zero errors.
3. **Tests pass.** `jest --passWithNoTests` exits with code 0.
4. **Expo export succeeds.** `npx expo export --platform web` exits with code 0 (web export as sandbox-compatible build check).
5. **No `any` in navigation types.** The `RootStackParamList` type must not contain `any`.
6. **Backend tests pass.** `pytest backend/tests/ -v` exits with code 0.
7. **Offline mode documented.** README includes a section describing offline capabilities and sync behavior.

## ADR Templates

### ADR-001: State Management Architecture

**Status:** Accepted

**Context:** The app requires managing UI state, server cache, and global app state. Using a single state management library for all three creates an overly complex store. Using three separate libraries adds bundle size.

**Decision:** Three-layer state management: `useState`/`useReducer` for local component state (zero bundle cost). `@tanstack/react-query` for server state (cache, sync status, background refetch). `zustand` for global client state (auth, user preferences). Each layer is documented with usage guidelines.

**Consequences:** Each library is best-of-class for its specific concern. The total bundle overhead is ~30KB gzipped. Developers have clear guidance on which layer to use for each type of state.

### ADR-002: Offline Sync Strategy

**Status:** Accepted

**Context:** The app must work offline and sync changes when connectivity is restored. We need a conflict resolution strategy when the same record is modified both locally and on the server during an offline period.

**Decision:** Optimistic local writes with last-write-wins conflict resolution on sync. Each record has a `updated_at` timestamp. During sync, the record with the later `updated_at` wins. For complex scenarios (field-level merging), the server is the authoritative source and local changes are rebased on top.

**Consequences:** Simple implementation that works well for personal apps (one user, one device dominant). For collaborative scenarios requiring per-field merge, we document the CRDT-based upgrade path (Automerge or Yjs) in the ADR.

### ADR-003: Navigation Pattern

**Status:** Accepted

**Context:** React Native requires a navigation library. Options include `react-navigation` (stack/tab/drawer navigator) and `expo-router` (file-based routing, similar to Next.js).

**Decision:** `expo-router` for all Expo projects. File-based routing reduces navigation boilerplate and makes route structure immediately visible to reviewers. TypeScript type safety via `expo-router`'s typed routes feature.

**Consequences:** Route names match the file system, eliminating navigator registration boilerplate. Deep linking is automatically configured. The constraint is that `expo-router` requires Expo SDK — not usable in bare React Native without Expo.

## Common Failure Modes

1. **Metro bundler crashes on native module.** Some npm packages with native code require `expo install` rather than `npm install` to get the Expo-compatible version. Fix: always use `expo install <package>` for packages with native modules.

2. **SQLite schema migration not run on upgrade.** When the app is updated with a new database schema, existing users' local databases are not migrated. Fix: implement a `runMigrations()` function on app startup that checks the current schema version and runs pending migrations.

3. **Platform-specific crash on iOS due to missing entitlement.** Features like push notifications, biometrics, and health data require entitlements in `app.json`'s `ios.entitlements`. Fix: add all required entitlements to `app.json` and test on both iOS and Android.

4. **React Query stale-while-revalidate causes flash of old data.** On navigation back to a screen, cached data shows briefly before fresh data loads. Fix: set appropriate `staleTime` (how long data is considered fresh) and use the `initialData` pattern for navigation params.

5. **Bundle includes sensitive config values.** Environment variables embedded in the bundle via `babel-plugin-transform-inline-environment-variables` are visible in the extracted bundle. Fix: never embed API keys or secrets in the bundle. Use Expo EAS Secrets or runtime environment injection for sensitive values.
