## Overview

Real-time systems skill files generate projects that demonstrate mastery of event-driven architectures, streaming data pipelines, and low-latency communication patterns. These projects showcase the ability to design systems where data flows continuously rather than in request-response batches — a critical skill for modern backend engineering. Generated projects include producer/consumer implementations, backpressure handling, at-least-once delivery guarantees, and WebSocket connection management. All projects are designed to run against an in-process Redis or in-memory broker so the sandbox can demonstrate end-to-end flow without requiring external infrastructure.

## Architecture Patterns

All real-time projects generated by this skill MUST follow these patterns:

1. **Backpressure is explicit.** Every producer-consumer pipeline has a defined strategy for when the consumer is slower than the producer: bounded queues with drop-oldest or block policy, explicit flow control signals, or circuit breakers. The strategy is documented in the ADR.

2. **Connection lifecycle is fully handled.** WebSocket handlers explicitly manage connect, message, error, and disconnect events. Reconnection logic (exponential backoff with jitter) is implemented on the client. Server-side cleanup happens on disconnect.

3. **Messages are versioned.** All event schemas include a `schema_version` field. The consumer checks the version and can handle at least two versions simultaneously to support rolling upgrades.

4. **At-least-once delivery is default, idempotency is required.** Message brokers are configured for at-least-once delivery. Consumers are idempotent: processing the same message twice produces the same result as processing it once (deduplication by message ID).

5. **Ordering guarantees are documented.** The system documents whether messages are ordered (per-partition, globally, or not at all) and the code reflects that reality. No code assumes global ordering unless the architecture guarantees it.

6. **Hot paths avoid I/O.** Critical real-time message processing paths (latency < 10ms) contain no blocking I/O, no ORM calls, and no external HTTP requests. All I/O is offloaded to async background tasks.

## Quality Criteria

The QualityGate checks the following for real-time projects:

1. **Tests pass.** `pytest tests/ -v` exits with code 0.
2. **Linting passes.** `ruff check .` reports zero errors.
3. **Type checking passes.** `mypy --strict src/` exits with code 0.
4. **WebSocket handler test.** At least one test simulates a full connect → send → receive → disconnect lifecycle using `pytest-asyncio` and an in-memory WebSocket mock.
5. **Backpressure test.** At least one test verifies that the queue bounded size is respected when the producer outpaces the consumer.
6. **No blocking calls in async handlers.** `ruff` or a custom lint rule must detect synchronous blocking calls (`time.sleep`, `requests.get`) inside `async def` handlers.
7. **Message schema versioning is present.** All message Pydantic models must include a `schema_version: Literal["1"]` field.

## ADR Templates

### ADR-001: Message Broker Selection

**Status:** Accepted

**Context:** The system requires a message broker for decoupling producers and consumers. Candidates are Redis pub/sub, Redis Streams, Kafka, and NATS.

**Decision:** Redis Streams for durability + consumer groups. Redis pub/sub for ephemeral broadcasts (e.g., WebSocket fan-out). Kafka is the production gold standard but requires ZooKeeper/KRaft and is operationally heavy for a portfolio demonstration. Redis Streams provides durable, ordered message storage with consumer groups and supports replay — covering 80% of Kafka use cases with a single dependency.

**Consequences:** We get durability, consumer groups, and message replay with minimal operational overhead. Maximum throughput (~1M msg/sec) is sufficient for portfolio demonstration. We document the Kafka migration path for production scenarios requiring multi-region replication.

### ADR-002: Backpressure Strategy

**Status:** Accepted

**Context:** When consumers are slower than producers, the system must degrade gracefully. Options are: block the producer, drop new messages, drop old messages, or apply rate limiting upstream.

**Decision:** Bounded in-memory queue (maxsize=10,000) with `drop_oldest` policy on overflow. The producer emits a `backpressure_applied` metric when the queue reaches 80% capacity. The API returns HTTP 429 to upstream clients when the queue is full.

**Consequences:** We never accumulate unbounded memory. Old messages are dropped in preference to new ones (inverse-recency). Upstream clients receive explicit 429 signals rather than silent drops, enabling them to implement backoff.

### ADR-003: WebSocket Connection Management

**Status:** Accepted

**Context:** WebSocket connections are long-lived and can be disrupted by network events. We need strategies for reconnection, message ordering during reconnect, and server-side resource cleanup.

**Decision:** Server: maintain a connection registry with heartbeat (ping every 30s). Clean up stale connections after 2 missed pings. Client: exponential backoff reconnection (1s, 2s, 4s, max 30s) with jitter. Message replay: client sends the last received message ID on reconnect; server replays missed messages from Redis Streams.

**Consequences:** Clients automatically recover from transient network issues. Message replay ensures no events are lost during brief disconnections. Long disconnections (>TTL of stream messages) result in a clean reconnect without replay.

## Common Failure Modes

1. **asyncio event loop blocking.** A single blocking call inside an async handler (e.g., `requests.get`, `time.sleep`) blocks the entire event loop, causing all WebSocket connections to freeze. Fix: use `asyncio.run_in_executor` for CPU-bound work and `httpx.AsyncClient` for HTTP calls.

2. **Redis pub/sub message loss on reconnect.** Redis pub/sub does not persist messages — a subscriber that disconnects misses messages published during the disconnect. Fix: use Redis Streams with consumer groups for durability, not pub/sub.

3. **WebSocket handler not cleaning up on disconnect.** If a client disconnects mid-message, the handler raises `WebSocketDisconnect`. Unhandled, this leaves connection state in the registry. Fix: wrap the handler body in a `try/finally` block that always removes the connection from the registry.

4. **Race condition in presence tracking.** When two clients disconnect simultaneously, both may attempt to update the same presence record. Fix: use Redis atomic operations (`SREM`, `SMEMBERS`) rather than read-modify-write.

5. **Message ordering assumptions broken by concurrent consumers.** Multiple consumer instances in a consumer group process messages concurrently, violating per-key ordering. Fix: use partition-key-based routing to ensure all messages for a given key go to the same consumer instance.
