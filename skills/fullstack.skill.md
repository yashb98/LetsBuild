## Overview

Full-stack web application skills generate end-to-end projects that demonstrate mastery across the entire web development stack: frontend UI, backend API, database design, authentication, and deployment configuration. These projects are the most universally applicable portfolio pieces because nearly every engineering role interacts with web applications. A well-crafted fullstack project proves that the candidate can reason about system boundaries, manage data flow between layers, handle authentication and authorization, and deliver a polished user experience backed by a robust API.

Projects generated by this skill always include a clear separation between frontend and backend, a relational database with migrations, proper error handling, and CI/CD configuration. They are designed to mirror the architecture patterns that hiring managers expect to see in production codebases.

## Architecture Patterns

All fullstack projects generated by this skill MUST follow these patterns:

1. **Separate API and Frontend layers.** The backend is a standalone FastAPI (or Express) service. The frontend is a standalone React/Next.js application. They communicate exclusively via a defined REST or GraphQL API. No server-side template rendering mixing concerns.

2. **Database migration system.** Every project includes Alembic (Python) or Knex (Node) migrations. The database schema is never created via raw SQL scripts alone.

3. **Authentication middleware.** JWT-based auth with refresh tokens for SPAs, or session-based auth with CSRF protection. Auth logic lives in dedicated middleware, not scattered across route handlers.

4. **Input validation at every boundary.** Pydantic models validate API inputs. Frontend forms validate before submission. Database constraints provide the final safety net.

5. **Environment-based configuration.** All secrets and environment-specific values come from environment variables. A `.env.example` file documents every required variable with placeholder values.

6. **Error handling contract.** The API returns structured error responses with consistent shape: `{"error": {"code": str, "message": str, "details": dict | null}}`. The frontend has a global error boundary and per-request error handling.

7. **API versioning.** Routes are prefixed with `/api/v1/`. This is non-negotiable for senior and staff projects.

## Quality Criteria

The QualityGate checks the following for fullstack projects:

1. **API tests pass.** `pytest` exits with code 0. At least one test per route.
2. **Frontend builds.** `npm run build` exits with code 0 with no TypeScript errors.
3. **Linting passes.** `ruff check .` (backend) and `eslint .` (frontend) report zero errors.
4. **Database migrations run.** `alembic upgrade head` succeeds against a clean database.
5. **No hardcoded secrets.** No API keys, tokens, or passwords in source files. `.env.example` uses placeholders only.
6. **README completeness.** README includes: project description, setup instructions, API documentation or link, environment variable reference, and architecture overview.
7. **Separation of concerns.** Backend and frontend are independently runnable. No import cycles between layers.
8. **Error responses are structured.** API error responses follow the standard error contract.

## ADR Templates

### ADR-001: API Framework Choice

**Status:** Accepted

**Context:** We need a Python web framework for the backend API. The primary candidates are FastAPI, Django REST Framework, and Flask.

**Decision:** We chose FastAPI because it provides automatic OpenAPI documentation, native async support, Pydantic-based request validation, and excellent performance. For a portfolio project, the auto-generated docs are especially valuable because reviewers can immediately explore the API.

**Consequences:** We gain type-safe request handling, automatic docs at `/docs`, and async-first design. We lose Django's built-in admin panel and ORM, which we replace with SQLAlchemy and Alembic.

### ADR-002: Database Selection

**Status:** Accepted

**Context:** The application needs a relational database for structured data with relationships (users, resources, permissions).

**Decision:** PostgreSQL with SQLAlchemy ORM and Alembic migrations. PostgreSQL is the industry standard for relational workloads, and SQLAlchemy provides a mature, well-documented ORM that maps cleanly to Pydantic models.

**Consequences:** We get ACID compliance, powerful query capabilities, and excellent tooling. The migration workflow via Alembic ensures reproducible schema changes. Local development uses Docker Compose to run PostgreSQL without requiring a system install.

### ADR-003: Authentication Strategy

**Status:** Accepted

**Context:** The application requires user authentication. Options include session-based auth, JWT tokens, and OAuth2 delegation.

**Decision:** JWT access tokens with short expiry (15 minutes) and HTTP-only refresh tokens (7 days). This approach is stateless on the server side and works well with SPA frontends that make API calls from the browser.

**Consequences:** We get stateless auth that scales horizontally. We must handle token refresh logic on the frontend and implement token revocation via a Redis blocklist for logout.

## Common Failure Modes

1. **CORS misconfiguration.** The most frequent sandbox failure. The FastAPI backend must explicitly allow the frontend origin in its CORS middleware. Fix: ensure `CORSMiddleware` is added with `allow_origins` matching the frontend dev server URL.

2. **Database connection string mismatch.** Alembic and the application sometimes use different connection string formats. Fix: both must read from the same `DATABASE_URL` environment variable.

3. **TypeScript strict mode errors in frontend.** Generated React components sometimes have implicit `any` types that fail `tsc --strict`. Fix: ensure all component props and state are explicitly typed.

4. **Missing `__init__.py` files.** Python package imports fail silently when `__init__.py` is missing from subdirectories. Fix: the Coder agent must create `__init__.py` in every Python package directory.

5. **Circular imports between models and schemas.** SQLAlchemy models importing Pydantic schemas and vice versa creates import cycles. Fix: use a `schemas/` directory that only imports from `models/`, never the reverse.

6. **Frontend API client not matching backend routes.** Route paths or HTTP methods drift between frontend API client and backend route definitions. Fix: generate the frontend API client from the OpenAPI spec, or validate routes in integration tests.
