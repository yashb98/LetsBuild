## Overview

ML Pipeline skills generate end-to-end machine learning projects that demonstrate the full lifecycle: data ingestion, feature engineering, model training, experiment tracking, evaluation, and serving via an API. These projects are the highest-signal portfolio pieces for ML engineer roles because they prove the candidate can move beyond Jupyter notebooks and build production-grade ML systems with reproducible training, versioned models, and monitored inference endpoints.

Every project generated by this skill includes a clear separation between training and serving code, experiment tracking with MLflow or equivalent, model versioning, structured configuration, and containerized deployment. The goal is to mirror the infrastructure that ML teams maintain in production environments.

## Architecture Patterns

All ML pipeline projects generated by this skill MUST follow these patterns:

1. **Separate training and serving code.** The training pipeline and the inference API are independent modules. Training code never imports from the API server, and the API server loads serialized model artifacts, not training scripts.

2. **Experiment tracking from day one.** Every training run logs parameters, metrics, and artifacts to MLflow (or equivalent). Model artifacts include version metadata and the exact configuration used to produce them.

3. **Configuration-driven pipelines.** All hyperparameters, data paths, and feature definitions live in YAML or TOML config files. No magic numbers in code. Training runs are fully reproducible from a config file plus a data snapshot.

4. **Model versioning and registry.** Trained models are versioned with semantic meaning (experiment ID, run ID, stage). The serving layer loads models by version reference, enabling rollback and A/B testing.

5. **Health and readiness endpoints.** The inference API exposes `/health` (is the server up?) and `/ready` (is a model loaded and warm?) endpoints. These are required for container orchestration.

6. **Structured prediction responses.** API responses include the prediction, model version, confidence score, and processing latency. Never return raw tensors or unstructured outputs.

7. **Data validation at ingestion.** Input data is validated against expected schemas before entering the training pipeline or inference endpoint. Schema violations produce structured errors, not silent failures.

## Quality Criteria

The QualityGate checks the following for ML pipeline projects:

1. **Training tests pass.** `pytest training/tests/ -v` exits with code 0. At least one test validates data loading, one validates model forward pass, and one validates metric computation.
2. **Serving tests pass.** `pytest serving/tests/ -v` exits with code 0. At least one test validates the prediction endpoint with mock model artifacts.
3. **Linting passes.** `ruff check .` reports zero errors across both training and serving code.
4. **No hardcoded secrets.** No API keys, tokens, or credentials in source files.
5. **Configuration is externalized.** All hyperparameters, data paths, and model parameters are defined in config files, not inline in code.
6. **Model artifacts are not committed.** `.gitignore` excludes `*.pt`, `*.pth`, `*.pkl`, `*.joblib`, `mlruns/`, and `artifacts/`.
7. **README includes reproducibility instructions.** README documents how to: install dependencies, configure, run training, run inference, and interpret experiment results.
8. **Health endpoints exist.** The serving API responds to `/health` and `/ready` with appropriate status codes.

## ADR Templates

### ADR-001: Model Framework Choice

**Status:** Accepted

**Context:** We need a deep learning framework for model development and training. The primary candidates are PyTorch, TensorFlow, and JAX.

**Decision:** We chose PyTorch because it offers the most intuitive debugging experience (eager execution by default), has the largest ecosystem of pre-trained models via Hugging Face, and is the dominant framework in ML research. For portfolio projects, PyTorch code is more readable to reviewers and hiring managers.

**Consequences:** We gain a mature ecosystem, excellent debugging, and broad community support. We lose TensorFlow's built-in serving infrastructure (TF Serving), which we replace with a custom FastAPI serving layer that gives us more control over pre/post-processing.

### ADR-002: Experiment Tracking Selection

**Status:** Accepted

**Context:** We need a system to track training experiments, compare runs, and version model artifacts.

**Decision:** MLflow for experiment tracking and model registry. MLflow is open-source, self-hostable, and provides a clean Python API for logging parameters, metrics, and artifacts. Its model registry supports stage transitions (staging, production, archived).

**Consequences:** We get reproducible experiments with full parameter and metric history. The MLflow UI provides visual comparison of runs. We accept the overhead of running an MLflow tracking server (mitigated by using file-based storage for local development).

### ADR-003: Training-Serving Separation

**Status:** Accepted

**Context:** The system must support both model training (batch, GPU-intensive) and model serving (real-time, latency-sensitive). These have fundamentally different compute requirements.

**Decision:** Separate the codebase into `training/` and `serving/` packages with independent dependencies. The only shared contract is the model artifact format (serialized PyTorch model + metadata JSON).

**Consequences:** Training and serving can be deployed, scaled, and updated independently. New model versions are published as artifacts and loaded by the serving layer without code changes. The tradeoff is maintaining a clear artifact contract between the two packages.

## Common Failure Modes

1. **PyTorch version mismatch in sandbox.** The sandbox base image may have a different PyTorch version than specified in `pyproject.toml`. Fix: pin exact PyTorch version in dependencies and ensure the sandbox image includes matching CUDA/CPU wheels.

2. **MLflow tracking URI not configured.** Tests fail because MLflow defaults to writing to `./mlruns/` and the working directory is not writable or unexpected. Fix: set `MLFLOW_TRACKING_URI` to a temp directory in test fixtures.

3. **Model artifact path hardcoding.** Training scripts that save models to hardcoded paths break in the sandbox. Fix: all artifact paths must come from configuration or environment variables.

4. **Missing test fixtures for model artifacts.** Serving tests need a pre-trained model artifact to load. Fix: include a tiny mock model fixture in `serving/tests/fixtures/` or generate one in `conftest.py`.

5. **Import errors between training and serving.** Accidentally importing training dependencies in serving code bloats the serving container and creates unnecessary dependencies. Fix: each package has its own `pyproject.toml` with independent dependency lists.

6. **Flaky tests due to random seeds.** Training tests that depend on model output values fail intermittently without fixed random seeds. Fix: set `torch.manual_seed()`, `numpy.random.seed()`, and `random.seed()` in test fixtures.
