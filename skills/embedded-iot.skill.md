## Overview

Embedded systems and IoT skill files generate projects that span the full IoT stack: device simulation or firmware stubs, MQTT-based telemetry ingestion, time-series storage, anomaly detection, and operational dashboards. Because the sandbox cannot run actual embedded hardware, projects use software-simulated devices (Python-based device simulators) that publish realistic telemetry over MQTT to the same backend that would receive real hardware. This demonstrates the complete system design while remaining fully testable in CI. Projects include firmware documentation, protocol schemas, over-the-air (OTA) update stubs, and fleet management APIs.

## Architecture Patterns

All embedded/IoT projects generated by this skill MUST follow these patterns:

1. **Device simulation mirrors real protocol behavior.** The Python device simulator uses the same MQTT topic structure, message schemas, and QoS settings that a real embedded device would use. Switching from simulated to real hardware requires changing only the device endpoint configuration.

2. **MQTT topic structure is hierarchical and documented.** Topics follow the pattern `{tenant}/{site}/{device_id}/{metric}`. Topic structure is documented in `docs/mqtt-schema.md`. Wildcards (`+`, `#`) in subscriptions are documented with their intended scope.

3. **Message schemas are versioned.** All telemetry messages include a `schema_version` field. The ingestion pipeline validates incoming messages against the schema and routes schema-mismatched messages to a dead-letter topic.

4. **Time-series data has explicit retention policy.** Raw telemetry has a 30-day retention. Downsampled 1-hour aggregates have 1-year retention. The retention policy is configured in the database setup script, not hardcoded in application logic.

5. **Device state is separated from telemetry.** Device identity (ID, firmware version, capabilities, last_seen) is stored in a relational database. Telemetry (time-series values) is stored in InfluxDB or TimescaleDB. The two are linked by device ID.

6. **Security: Device authentication and TLS.** MQTT connections use TLS with certificate-based device authentication in production. The sandbox uses username/password auth for simplicity, with the production TLS configuration documented in the deployment guide.

## Quality Criteria

The QualityGate checks the following for embedded/IoT projects:

1. **Tests pass.** `pytest tests/ -v` exits with code 0.
2. **Linting passes.** `ruff check .` reports zero errors.
3. **Type checking passes.** `mypy --strict .` (Python components) exits with code 0.
4. **MQTT schema document exists.** `docs/mqtt-schema.md` must document all topics with their payload schema and QoS level.
5. **Device simulator runs and publishes.** `python -c "from device_simulator.simulator import DeviceSimulator; ..."` completes without error.
6. **Message schema validation is tested.** At least one test verifies that a malformed message is routed to the dead-letter topic rather than crashing the ingestion pipeline.
7. **Retention policy is configured.** Database setup script must include retention policy configuration.

## ADR Templates

### ADR-001: MQTT Broker Selection

**Status:** Accepted

**Context:** We need an MQTT broker for device telemetry. Options include Mosquitto (self-hosted), HiveMQ, AWS IoT Core, and EMQX.

**Decision:** Mosquitto for development and portfolio demonstration. It is the most widely used open-source MQTT broker, runs in a single Docker container, and is configuration-free for basic use. For production, AWS IoT Core or HiveMQ is recommended (and documented in the deployment guide) for managed TLS, device registry, and horizontal scaling.

**Consequences:** Zero operational overhead for portfolio demonstration. The Mosquitto configuration in `docker-compose.yml` mirrors the TLS+auth configuration that production brokers require, making the upgrade path minimal.

### ADR-002: Time-Series Storage Selection

**Status:** Accepted

**Context:** Telemetry data has high write throughput and time-range query patterns that are poorly served by row-oriented databases.

**Decision:** SQLite with time-indexed tables for the portfolio demonstration (zero external dependencies). The ingestion pipeline is designed with a `TimeSeriesStore` abstraction that can be swapped for InfluxDB or TimescaleDB without changing application logic. The abstraction is documented and the InfluxDB implementation is provided as an alternative module.

**Consequences:** The sandbox runs without InfluxDB as a dependency. The `TimeSeriesStore` abstraction ensures that upgrading to a real time-series database (for production or for a hiring manager who wants to run the full stack) requires only a configuration change.

### ADR-003: Device Authentication Model

**Status:** Accepted

**Context:** In production, devices must authenticate before publishing telemetry. For the portfolio demonstration, we must balance security correctness with sandbox operability.

**Decision:** Username/password MQTT authentication in the sandbox (username = device_id, password = hashed pre-shared key). In production, TLS mutual authentication with per-device certificates signed by a private CA. The production authentication flow is fully documented in `docs/deployment-guide.md`.

**Consequences:** The sandbox is fully functional without a CA. The security model is honest about its limitations (username/password is weaker than mTLS) and documents the production upgrade path explicitly.

## Common Failure Modes

1. **MQTT client reconnect loop.** If the broker is not running when the client starts, naive implementations retry immediately in a tight loop. Fix: implement exponential backoff reconnection in the MQTT client (1s, 2s, 4s, max 60s) with a maximum retry count before alerting.

2. **Time-series query returns empty results due to timezone mismatch.** Timestamps stored as UTC but queried with local timezone offsets return empty or incorrect ranges. Fix: store all timestamps as UTC, enforce UTC in `datetime` objects (`datetime.now(UTC)`), and document the timezone policy in code comments.

3. **Message ordering violations at QoS 0.** MQTT QoS 0 (at-most-once) does not guarantee delivery or ordering. Telemetry processed out of order can produce incorrect rate-of-change calculations. Fix: use QoS 1 (at-least-once) for telemetry and include a sequence number in the message payload for ordering validation.

4. **Device simulator message rate overwhelming the ingestion pipeline.** A simulator generating 1000 messages/second can overwhelm a synchronous ingestion pipeline. Fix: the ingestion pipeline must use async MQTT callbacks and a bounded queue (maxsize=10,000) with backpressure signaling to the simulator.

5. **Schema version mismatch breaks ingestion silently.** If a new device firmware publishes messages in a new schema version that the ingestion pipeline does not recognize, messages may be silently discarded or incorrectly parsed. Fix: the ingestion pipeline must log a warning and route unrecognized schema versions to a dead-letter topic, never silently drop them.
