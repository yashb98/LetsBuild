## Overview

Autonomous agent and robotics skill files generate projects that showcase self-directed, goal-oriented systems: multi-agent orchestration frameworks, reinforcement learning environments, robotic task planners, and LLM-powered autonomous executors. These projects are the most technically ambitious in the LetsBuild catalog — they combine systems programming, AI reasoning, and real-time control. Generated projects use the canonical stop_reason agentic loop pattern from the LetsBuild architecture (not iteration caps), demonstrate proper tool scoping (≤5 tools per agent), and include explicit policy gates to prevent runaway execution.

## Architecture Patterns

All autonomous agent projects generated by this skill MUST follow these patterns:

1. **Stop_reason agentic loop (mandatory).** Every agent implements the canonical loop: send request → check `stop_reason` → if `tool_use`, execute tools and continue → if `end_turn`, extract result. NEVER use iteration caps as the primary stop condition.

2. **Tool scoping (≤5 tools per agent).** Each agent's `tools` list contains at most 5 tools, scoped to its role. A Reviewer agent never has `write_file`. A Planner never has `bash_execute`. Scoping is enforced at initialization.

3. **Independent reviewer pattern.** When a review step is required, the Reviewer agent is instantiated with a fresh API client and receives no conversation history from the Coder/Actor. It receives only the artifact and a quality checklist.

4. **Explicit policy gates.** All irreversible actions (delete file, send email, commit code, call external API with state mutation) require passing a deterministic `PolicyGate` check before execution. The gate is pure Python code, not a prompt instruction.

5. **Structured tool results.** Every tool implementation returns a typed result: `{"status": "success" | "error", "data": ..., "error_category": ..., "is_retryable": ...}`. Agents never parse free-text tool output.

6. **Context budget management.** Every agent tracks approximate token usage and trims old tool outputs when approaching the context limit. Key facts are preserved in a structured "case facts" block at the start of context.

## Quality Criteria

The QualityGate checks the following for autonomous agent projects:

1. **Tests pass.** `pytest tests/ -v` exits with code 0.
2. **Agentic loop uses stop_reason.** A lint check or AST inspection verifies no integer iteration cap is used as the primary loop termination in agent files.
3. **Policy gate tests include blocking cases.** `test_policy_gate.py` must include at least one test where the gate blocks an action and one where it permits it.
4. **Tool result structure is typed.** All tool functions return a Pydantic model with `status`, `data`, `error_category`, and `is_retryable` fields.
5. **No API keys in prompts.** System prompts and user messages must not contain API key patterns.
6. **Linting and type checking pass.** `ruff check .` and `mypy --strict src/` both exit with code 0.

## ADR Templates

### ADR-001: Agent Topology

**Status:** Accepted

**Context:** The system requires multiple specialized agents. We must choose between hierarchical (Planner → Executors → Reviewer), mesh (peer-to-peer message passing), and sequential topologies.

**Decision:** Hierarchical topology with a Coordinator agent that dispatches tasks to specialized sub-agents. Each sub-agent completes its assigned task and returns a structured result to the Coordinator. The Coordinator synthesizes results and decides on next steps.

**Consequences:** Clear authority hierarchy simplifies reasoning about state. The Coordinator is the single point of failure but also the single point of control — it can implement budget tracking, retry logic, and policy enforcement. Sub-agent parallelism is possible when tasks are independent.

### ADR-002: Tool Scoping Policy

**Status:** Accepted

**Context:** Each agent needs access to tools to accomplish its role. Giving agents too many tools increases the risk of unintended actions and makes behavior harder to audit.

**Decision:** Maximum 5 tools per agent, strictly scoped to the agent's role. Planner: `search`, `read_document`, `list_resources`. Executor: `write_file`, `execute_command`, `read_file`. Reviewer: `read_file`, `list_directory`. Policy enforcement is implemented as a pre-tool-call check, not a prompt instruction.

**Consequences:** Agents cannot take actions outside their designated scope even if the LLM produces a tool call for an out-of-scope tool. The pre-call check raises a `ToolScopeViolation` error that is logged and returned as a structured error to the agent.

### ADR-003: Irreversible Action Policy

**Status:** Accepted

**Context:** Some agent actions are irreversible (delete files, send notifications, publish to external APIs). We must prevent the agent from taking these actions without appropriate oversight.

**Decision:** All irreversible actions go through `PolicyGate.check_action(action_type, payload)` before execution. The gate evaluates deterministic rules: allow list of permitted action types, payload schema validation, and (for staff-level projects) a human-approval queue for high-risk actions.

**Consequences:** The LLM cannot override policy gates regardless of what its prompt or reasoning says. This is a core LetsBuild principle: business rules enforced by code, not prompts.

## Common Failure Modes

1. **Iteration cap used instead of stop_reason.** Using `for i in range(50): ...` as the primary agent loop termination is a common mistake. Fix: the loop must be `while True:` with `break` only on `response.stop_reason == "end_turn"`.

2. **Context accumulation causing token limit errors.** Long agentic runs accumulate tool outputs that fill the context window. Fix: implement `PostToolUse` trimming that replaces verbose tool outputs with a 2-3 line summary after they are processed.

3. **Tool calling with wrong argument types.** The LLM may call a tool with a string argument where an integer is expected. Fix: validate all tool input arguments against the declared JSON schema before execution; return a structured validation error instead of raising an exception.

4. **Sub-agents sharing context unintentionally.** When sub-agents are implemented as methods that share `self.messages`, reviewer context bleeds into coder context. Fix: each sub-agent instantiation gets its own `messages: list[dict] = []`.

5. **Runaway agent loop on tool errors.** If a tool consistently returns errors, the agent may loop indefinitely trying different approaches. Fix: implement a per-tool retry budget (max 3 attempts) that the Coordinator tracks and uses to abort the task if exceeded.
