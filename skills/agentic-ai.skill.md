## Overview

Agentic AI skills generate projects that demonstrate the ability to build autonomous systems where LLM-powered agents use tools, maintain context, collaborate with other agents, and accomplish complex tasks with minimal human intervention. These are the highest-signal portfolio pieces for AI engineering roles because the field is evolving rapidly and most candidates can only show basic prompt engineering or simple chatbot wrappers.

A well-crafted agentic AI project proves the candidate understands tool-use patterns, context engineering, agent orchestration, structured output extraction, error recovery, and the critical distinction between what should be handled by deterministic code versus what should be delegated to an LLM. Every project generated by this skill includes proper agent loops driven by `stop_reason` (never iteration caps), scoped tool sets, structured error handling, and comprehensive testing with mocked LLM responses.

## Architecture Patterns

All agentic AI projects generated by this skill MUST follow these patterns:

1. **stop_reason-driven agent loops.** Agent loops continue based on the model's `stop_reason`, not arbitrary iteration caps. The loop processes `tool_use` responses by executing tools and appending results, and exits on `end_turn`. A safety cap of 50 turns is acceptable as a fallback only.

2. **Scoped tool sets (maximum 5 per agent).** Each agent receives only the tools relevant to its specialization. A research agent gets search and read tools. A code agent gets file manipulation tools. No agent gets cross-domain tools.

3. **Structured output via tool_use.** All structured data extraction uses `tool_choice` with JSON schemas. Never parse free-text LLM output for structured data. Every tool returns typed Pydantic models.

4. **Independent review agents.** Any agent that evaluates or reviews output MUST operate with zero prior context from the generating agent. The reviewer receives only the artifact and evaluation criteria.

5. **Structured error handling.** Every tool execution returns structured errors with `errorCategory` (transient, validation, business, permission) and `isRetryable` boolean. Agents use this to decide whether to retry, skip, or escalate.

6. **Context engineering.** Implement PostToolUse trimming to prevent context window bloat. Long tool outputs are summarized before being appended to the conversation. Key findings are positioned early in the context.

7. **Deterministic policy enforcement.** Business rules, quality gates, and security checks are implemented as Python code (gates/hooks), never as prompt instructions. The LLM should not be trusted to enforce rules that must be 100% reliable.

## Quality Criteria

The QualityGate checks the following for agentic AI projects:

1. **All tests pass.** `pytest tests/ -v` exits with code 0. Tests MUST mock all LLM API calls -- no real API calls in tests.
2. **Agent loops use stop_reason.** Code review verifies that agent loops check `response.stop_reason`, not iteration counters, as the primary exit condition.
3. **Tool scoping is enforced.** Each agent class declares its `allowed_tools` list and the orchestrator validates tool calls against it.
4. **Structured output is used.** All tool schemas are defined as Pydantic models with `Field(description=...)` annotations. No free-text parsing for structured data.
5. **Linting passes.** `ruff check .` and `ruff format --check .` report zero errors.
6. **No hardcoded API keys.** No API keys, tokens, or credentials in source files. All keys come from environment variables.
7. **Error handling is structured.** Tool functions return typed error objects with category and retryability, not raw exception strings.
8. **README documents agent architecture.** README includes an agent interaction diagram, tool descriptions, and instructions for running with mocked vs. real LLM backends.

## ADR Templates

### ADR-001: LLM Provider Choice

**Status:** Accepted

**Context:** The agentic system needs an LLM backend for reasoning, tool use, and text generation. The primary candidates are Anthropic Claude, OpenAI GPT-4, and open-source models via vLLM.

**Decision:** We chose Anthropic Claude as the primary provider because it offers the most reliable tool_use implementation, supports structured output extraction via tool_choice forcing, and provides clear stop_reason signals that drive our agent loops. The stop_reason-based loop pattern (continue on "tool_use", exit on "end_turn") is more reliable than parsing completion text.

**Consequences:** We get robust tool_use with JSON schema validation, predictable agent loop behavior, and strong reasoning capabilities. We accept vendor dependency on Anthropic and mitigate it with an adapter pattern that allows swapping providers. Tests mock the API client, so they run without credentials.

### ADR-002: Vector Database Selection

**Status:** Accepted

**Context:** The system needs a vector database for semantic search over documents, past interactions, and learned patterns. Options include ChromaDB, Pinecone, Weaviate, and Qdrant.

**Decision:** ChromaDB for local development and testing, with an adapter interface supporting Pinecone or Weaviate for production deployments. ChromaDB is embedded (no server required), has a simple Python API, and supports persistent storage.

**Consequences:** We get zero-infrastructure vector search for development and testing. The adapter pattern allows production deployments to use managed vector databases. We accept that ChromaDB's performance characteristics differ from production databases and test accordingly.

### ADR-003: Agent Orchestration Pattern

**Status:** Accepted

**Context:** Multiple agents must collaborate to accomplish complex tasks. Options include: hierarchical (manager dispatches to workers), mesh (agents communicate peer-to-peer), sequential (pipeline), and ring (cyclic validation).

**Decision:** Mesh topology with a lightweight message bus. Agents communicate via structured messages through a central bus. Any agent can address messages to any other agent. The bus handles routing, logging, and dead letter queues.

**Consequences:** We get flexible agent communication that supports dynamic collaboration patterns. The message bus provides observability and debugging capabilities. We accept the complexity of managing message routing and potential message storms, mitigated by rate limiting and message TTLs.

## Common Failure Modes

1. **Infinite agent loops.** Agents that never receive `stop_reason == "end_turn"` loop forever. Fix: implement the safety cap of 50 turns as a fallback, log a warning, and return partial results rather than hanging.

2. **Context window overflow.** Agents that accumulate long tool outputs exhaust the context window. Fix: implement PostToolUse hooks that trim verbose outputs (e.g., truncate file contents to first 200 lines, summarize search results to top 5).

3. **Mock response format mismatch.** Tests that mock LLM responses with incorrect structure (missing `stop_reason`, wrong `content` format) cause spurious test failures. Fix: create a shared `MockResponse` factory in `conftest.py` that matches the real Anthropic response format exactly.

4. **Tool schema drift.** Tool function signatures and their JSON schema definitions get out of sync. Fix: generate tool schemas from Pydantic models using `model_json_schema()` rather than maintaining separate schema definitions.

5. **Non-deterministic agent behavior in tests.** Agent tests that depend on specific LLM reasoning paths are brittle. Fix: test the agent loop mechanics (tool execution, stop condition, error handling) rather than the semantic content of LLM responses.

6. **Vector store cold start.** Tests that depend on populated vector stores fail when the store is empty. Fix: populate test fixtures in `conftest.py` before tests run, and explicitly test the empty-store case as a valid scenario.
