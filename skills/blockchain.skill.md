## Overview

Blockchain skill files generate smart contract projects and decentralized applications that demonstrate practical Web3 engineering: writing secure Solidity contracts, testing them with Hardhat or Foundry, and building frontends that interact with deployed contracts via ethers.js or wagmi. Security is the dominant concern in this domain — every generated project includes explicit reentrancy guards, access control, and an audit checklist. All projects run against a local Hardhat network or Anvil fork so the sandbox requires no testnet connectivity. Projects include contract verification scripts, NatSpec documentation, and gas optimization reports.

## Architecture Patterns

All blockchain projects generated by this skill MUST follow these patterns:

1. **Security-first: Checks-Effects-Interactions.** Every state-mutating function follows the CEI pattern: validate inputs first (checks), update state second (effects), call external contracts last (interactions). This is the primary defense against reentrancy attacks.

2. **Access control is explicit.** Every privileged function uses OpenZeppelin `AccessControl` or `Ownable`. No unprotected state-mutating external functions. Role names are constants, not magic strings.

3. **Reentrancy guards on external calls.** Any function that calls an external contract must use `ReentrancyGuard` from OpenZeppelin. No exceptions.

4. **Integer arithmetic uses SafeMath or Solidity 0.8+.** All arithmetic is in Solidity 0.8.x (which reverts on overflow by default) or explicitly uses `SafeMath`. No unchecked blocks without documented justification.

5. **Events for all state changes.** Every significant state change emits an event. Events are the primary audit log and the data source for off-chain indexers (The Graph).

6. **NatSpec documentation on all public functions.** Every `public` and `external` function has NatSpec `@dev`, `@param`, and `@return` documentation. The `npx hardhat docgen` command generates readable API docs.

7. **Gas optimization is documented.** A `gas-report.txt` generated by `hardhat-gas-reporter` is committed to the repository. Any gas optimization decision (packed structs, custom errors, unchecked arithmetic) is documented in comments.

## Quality Criteria

The QualityGate checks the following for blockchain projects:

1. **Contract tests pass.** `npx hardhat test` exits with code 0 with ≥90% branch coverage.
2. **No reentrancy vulnerabilities.** Slither or manual audit checklist verifies CEI pattern and `ReentrancyGuard` on all external-call functions.
3. **Gas report exists.** `gas-report.txt` must be present and non-empty.
4. **NatSpec is complete.** All `public` and `external` functions have `@dev`, `@param` (if parameters exist), and `@return` (if return values exist) NatSpec comments.
5. **Audit checklist is completed.** `docs/audit-checklist.md` has all items checked, including reentrancy, access control, integer overflow, and front-running assessment.
6. **Frontend builds.** `cd frontend && npm ci && npm run build` exits with code 0.
7. **No private keys committed.** The `.gitignore` must exclude `.env`, and no private key patterns appear in source files.

## ADR Templates

### ADR-001: Contract Architecture

**Status:** Accepted

**Context:** We must choose between a monolithic contract and a proxy upgrade pattern. Monolithic is simpler and cheaper to deploy. Proxy patterns (UUPS, Transparent) allow upgrades but add complexity and trust assumptions.

**Decision:** Immutable contracts (no proxy) for this portfolio project. Proxy patterns are appropriate for production protocols where bugs must be fixable post-deployment. For a portfolio demonstration, immutable contracts are cleaner, cheaper to audit, and avoid the proxy-specific attack surface. An upgrade path using UUPS is documented in the ADR as the production next step.

**Consequences:** Contract logic cannot be changed after deployment. We mitigate this by using OpenZeppelin's audited contracts as the base. The README documents the specific OpenZeppelin version and the immutability rationale.

### ADR-002: Security Model

**Status:** Accepted

**Context:** Smart contracts are immutable once deployed and hold real value. Security must be designed-in, not bolted on. We need a documented security model that reviewers can audit.

**Decision:** The security model has four layers: (1) Solidity 0.8.x arithmetic safety, (2) OpenZeppelin contracts for access control and reentrancy, (3) CEI pattern enforced in code review, (4) Slither static analysis in CI. All external oracle and token contract interactions assume adversarial inputs.

**Consequences:** We accept that this is not a professional audit. The audit checklist in `docs/audit-checklist.md` documents what has been checked and what would require a professional audit before mainnet deployment.

### ADR-003: Front-End Wallet Integration

**Status:** Accepted

**Context:** The frontend must connect to user wallets. Options include direct `window.ethereum` injection, ethers.js `BrowserProvider`, wagmi, and Web3Modal.

**Decision:** wagmi v2 with Viem for type-safe contract interactions, and ConnectKit or RainbowKit for the wallet connection UI. wagmi provides React hooks that handle wallet connection state, network switching, and transaction lifecycle out of the box.

**Consequences:** Excellent developer experience and type-safe contract calls via `useReadContract` and `useWriteContract`. Bundle size is approximately 80KB gzipped. The tradeoff is lock-in to the wagmi/Viem ecosystem, which is acceptable given their market adoption.

## Common Failure Modes

1. **Reentrancy via external call before state update.** Writing `transfer(amount)` before `balances[msg.sender] -= amount` is the classic reentrancy vulnerability. Fix: always update state before external calls (CEI pattern). Use `ReentrancyGuard` as defense-in-depth.

2. **Front-running in auction mechanics.** On-chain auctions where the winning bid is visible in the mempool can be front-run by miners. Fix: implement a commit-reveal scheme where bids are committed as hashes, then revealed after the bidding period.

3. **Integer overflow in reward calculation.** Reward calculations involving multiplication before division can overflow even in Solidity 0.8.x if intermediate values exceed `uint256`. Fix: use a precision multiplier (e.g., `1e18`) and divide at the end, or use FullMath library for 512-bit intermediate precision.

4. **Hardhat test isolation failures.** Tests that depend on block numbers or timestamps may fail if test execution order changes. Fix: use `hardhat_mine` and `evm_setNextBlockTimestamp` to explicitly set block state before time-dependent tests.

5. **IPFS metadata not pinned.** NFT metadata stored on IPFS without pinning is garbage-collected after a few days. Fix: use Pinata or NFT.Storage API (with stubs in the sandbox) and document the pinning requirement in the deployment checklist.
