## Overview

API and backend service skill files generate production-grade REST and gRPC services that demonstrate deep understanding of API design, data modeling, caching strategies, and operational excellence. These projects are the bread-and-butter of backend engineering portfolios — the difference between a mediocre and an impressive API project lies in the details: proper versioning, pagination, rate limiting, idempotency, structured error responses, and observability. Generated projects include OpenAPI documentation, structured logging, distributed tracing stubs, and comprehensive test suites covering both unit and integration test patterns.

## Architecture Patterns

All API backend projects generated by this skill MUST follow these patterns:

1. **Versioned routes.** All routes are prefixed `/api/v1/`. A breaking change increments the version. Old versions remain accessible for at least one version cycle.

2. **Structured error responses.** Every error response follows the standard contract: `{"error": {"code": str, "message": str, "request_id": str, "details": dict | null}}`. HTTP status codes are never the only signal.

3. **Cursor-based pagination.** List endpoints use cursor-based pagination (not offset/limit) for stable results on large or frequently-updated datasets. Response includes `next_cursor` and `has_more` fields.

4. **Idempotency keys.** Mutating endpoints (`POST`, `PUT`, `DELETE`) accept an optional `Idempotency-Key` header. Duplicate requests with the same key return the original response from cache (Redis TTL: 24h).

5. **Request ID propagation.** Every request is assigned a `X-Request-Id` (UUID4) at the ingress middleware. It is returned in all responses and included in all log entries and outbound service calls.

6. **Health and readiness endpoints.** Every service exposes `/health` (liveness: returns 200 if the process is running) and `/ready` (readiness: returns 200 only if all dependencies — DB, Redis, downstream services — are reachable).

7. **Dependency injection for infrastructure.** Database sessions, Redis clients, and external service clients are injected via FastAPI `Depends()`. They are never imported as module-level singletons in route handlers.

## Quality Criteria

The QualityGate checks the following for API backend projects:

1. **Tests pass.** `pytest tests/ -v` exits with code 0. Coverage must be ≥80% for `app/`.
2. **Linting passes.** `ruff check .` reports zero errors.
3. **Type checking passes.** `mypy --strict app/` exits with code 0.
4. **OpenAPI spec is generated.** Running `python -c "from app.main import app; import json; print(json.dumps(app.openapi()))"` produces valid JSON with all routes documented.
5. **Pagination is cursor-based.** List endpoints must not use `offset` parameter. Tests verify cursor continuity across pages.
6. **Health endpoints respond.** `/health` and `/ready` both return 200 with valid JSON.
7. **Alembic migrations run clean.** `alembic upgrade head` succeeds against a fresh SQLite database in offline mode.

## ADR Templates

### ADR-001: Pagination Strategy

**Status:** Accepted

**Context:** List endpoints must support pagination. The two common approaches are offset/limit pagination and cursor-based pagination.

**Decision:** Cursor-based pagination using an opaque base64-encoded cursor (encoding the sort key value + primary key). Offset pagination is simpler to implement but produces inconsistent results when the dataset is mutated between pages (new records shift offsets).

**Consequences:** Cursor pagination provides stable results for clients paginating through live data. The tradeoff is that clients cannot jump to an arbitrary page number — they must traverse sequentially. This is acceptable for all planned use cases (infinite scroll, API consumers iterating datasets).

### ADR-002: Rate Limiting Design

**Status:** Accepted

**Context:** The API must enforce per-client rate limits to prevent abuse. Options include fixed-window, sliding-window, and token-bucket algorithms.

**Decision:** Sliding-window rate limiting using Redis with Lua scripts for atomic execution. Sliding windows avoid the burst-at-window-boundary problem of fixed-window algorithms. Lua scripts ensure the read-increment-expire sequence is atomic even under concurrent access.

**Consequences:** We get smooth rate limiting without the boundary burst problem. Redis adds a single round-trip per request for rate limit checking (~0.5ms). Rate limit headers (`X-RateLimit-Limit`, `X-RateLimit-Remaining`, `X-RateLimit-Reset`) are included in every response.

### ADR-003: Error Response Contract

**Status:** Accepted

**Context:** The API must return consistent, machine-readable error responses. Inconsistent error formats force API clients to handle each endpoint differently.

**Decision:** All error responses use the structure `{"error": {"code": str, "message": str, "request_id": str, "details": dict | null}}`. HTTP status codes follow RFC 7231. Validation errors include the specific field names in `details`. The `code` field uses a dot-namespaced string (e.g., `resource.not_found`, `validation.required_field`).

**Consequences:** Clients can build generic error handling on the `code` field without parsing human-readable messages. The `request_id` enables correlation with server logs. All FastAPI validation exceptions are caught by a global exception handler and reformatted to match this contract.

## Common Failure Modes

1. **N+1 query problem in list endpoints.** Fetching related objects in a loop generates N+1 queries. Fix: use SQLAlchemy's `selectinload` or `joinedload` for all relationships accessed in list responses.

2. **Alembic autogenerate missing columns.** Alembic `--autogenerate` may miss columns if models are not imported before `Base.metadata` is inspected. Fix: ensure all models are imported in `alembic/env.py` before the metadata comparison.

3. **Idempotency key not persisted before processing.** If the service persists the idempotency key after processing, a crash during processing leaves a window where a retry would reprocess. Fix: persist the idempotency key with `status=pending` before processing, then update to `status=complete` after.

4. **CORS headers missing on error responses.** FastAPI's CORS middleware only adds headers to successful responses in some configurations. Fix: add CORS middleware before exception handlers, or use a custom exception handler that adds headers explicitly.

5. **Cursor leaking internal IDs.** If cursors encode raw database primary keys, API consumers can infer total record counts or guess other records' IDs. Fix: encode the cursor as a base64-encoded JSON blob containing only the sort key values, not the primary key.
