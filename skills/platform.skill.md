## Overview

Platform engineering skill files generate infrastructure-as-code projects, internal developer platform tools, and observability systems that demonstrate the ability to design, automate, and operate scalable infrastructure. Platform projects are highly valued because they show candidates can reduce toil for engineering teams — building paved roads rather than one-off solutions. Generated projects include Terraform modules, Kubernetes manifests, CI/CD pipeline definitions, runbooks, and observability dashboards. All infrastructure targets AWS or GCP using abstracted resource definitions so the sandbox can validate syntax and policy without provisioning real cloud resources.

## Architecture Patterns

All platform projects generated by this skill MUST follow these patterns:

1. **Infrastructure is code.** All cloud resources are defined in Terraform or Pulumi. No clickops resources. Modules are parameterized and reusable. A `modules/` directory separates reusable components from environment-specific compositions.

2. **Everything is idempotent.** All scripts, controllers, and CLI commands can be run multiple times without side effects. State transitions are explicit and reversible.

3. **Drift detection is built in.** For GitOps and IaC tools, a `plan` command shows the difference between desired and actual state without making changes. A `apply` command converges state.

4. **Audit logs are append-only.** Every privileged action (secret access, cluster change, resource creation) writes a structured log entry with actor, action, resource, and timestamp. Audit logs are never deleted.

5. **Policy enforcement is declarative.** Access control and compliance rules are defined in policy files (OPA Rego, AWS SCP JSON), not in application code. Policy files are tested independently.

6. **Runbooks exist for every alert.** For observability projects, every Prometheus alert has a corresponding runbook in `docs/runbooks/` explaining symptoms, diagnosis steps, and remediation.

## Quality Criteria

The QualityGate checks the following for platform projects:

1. **Tests pass.** `pytest tests/ -v` exits with code 0.
2. **Linting passes.** `ruff check .` (Python) reports zero errors.
3. **Terraform validates.** `terraform init -backend=false && terraform validate` exits with code 0.
4. **Kubernetes manifests are valid.** `kubectl apply --dry-run=client -f k8s/base/` exits with code 0.
5. **Policy tests pass.** `opa test policies/ -v` exits with code 0 (if OPA policies are present).
6. **No secrets in IaC.** Terraform and Helm files must not contain hardcoded passwords, tokens, or keys. All sensitive values use `var.*` or secret references.
7. **README includes architecture diagram.** ASCII or Mermaid diagram showing component topology is required.

## ADR Templates

### ADR-001: IaC Toolchain Selection

**Status:** Accepted

**Context:** We need an infrastructure-as-code tool for provisioning cloud resources. Candidates are Terraform (HCL), Pulumi (Python/TypeScript), and AWS CDK (TypeScript).

**Decision:** Terraform with HCL for resource definitions. Terraform is the most widely adopted IaC tool in platform engineering roles. Its declarative syntax and state management model are well understood by hiring managers. Modules are organized using the standard registry module structure.

**Consequences:** We gain deterministic plan/apply workflows and a large ecosystem of community modules. We accept HCL's lack of real programming constructs (loops and conditionals are limited). Complex conditional logic is handled by wrapper Python scripts that generate tfvars.

### ADR-002: Observability Data Model

**Status:** Accepted

**Context:** The monitoring stack must collect metrics, logs, and traces. We need to decide on the data model and correlation strategy.

**Decision:** OpenTelemetry as the instrumentation standard for all services. Metrics to Prometheus (OTLP receiver), logs to Loki (structured JSON), traces to Tempo. Grafana as the unified query interface. Correlation via trace ID propagated in structured log fields.

**Consequences:** OpenTelemetry provides vendor-neutral instrumentation. Single trace ID across metrics/logs/traces enables root cause analysis across signals. Cardinality must be controlled: high-cardinality label values (user IDs, request IDs) are excluded from metric labels.

### ADR-003: GitOps Reconciliation Strategy

**Status:** Accepted

**Context:** The GitOps controller must reconcile Git state to cluster state. We must decide on the reconciliation interval, conflict resolution, and rollback strategy.

**Decision:** Level-triggered reconciliation every 60 seconds plus event-triggered on Git push webhooks. Conflict resolution: desired state (Git) always wins over actual state (cluster). Rollback: revert the Git commit and let the controller re-apply the previous state.

**Consequences:** We get eventual consistency with a maximum drift window of 60 seconds. Git history becomes the authoritative change log. Rollbacks are explicit commits, not ad-hoc `kubectl` commands.

## Common Failure Modes

1. **Terraform provider authentication fails in sandbox.** Cloud provider CLIs require credentials that are not present in the sandbox. Fix: use `terraform validate` (not `plan` or `apply`) in CI, which validates syntax and resource schema without making API calls.

2. **Kubernetes manifests reference missing Secrets.** Manifests that reference Kubernetes Secrets by name fail on `kubectl apply --dry-run` if the secrets don't exist. Fix: use `--dry-run=client` (client-side validation only) rather than `--dry-run=server`.

3. **OPA policy tests fail on import paths.** OPA's `import` statements use package paths, not file paths. Fix: organize policy files by package name and run `opa test` from the repository root, not from the `policies/` subdirectory.

4. **Helm chart values schema violations.** Generated Helm values files sometimes violate the chart's `values.schema.json`. Fix: run `helm lint` as a quality gate step and include it in the sandbox validation plan.

5. **Python reconciler loop busy-waits.** Naive reconcile loops that run `time.sleep(0)` in a tight loop consume 100% CPU. Fix: use `asyncio.sleep(interval)` between reconcile cycles and add jitter to prevent thundering herd on startup.
